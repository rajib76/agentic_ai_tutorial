Section 01

The Big Picture — Where Generative & Agentic AI Fit

"AI," "machine learning," "generative AI," "agents" — these words get thrown around as if they mean the same thing. They don't. Before we open the hood, let's draw the map so you always know what is a part of what — and answer a question most tutorials skip: why is it even called "generative"?

Nested fields: AI contains ML contains Generative AI

The cleanest way to picture this is as a set of nested circles, each one a more specialized slice of the one around it.

Reading from the outside in: Artificial Intelligence is the broad ambition of machines doing things that normally require human intelligence. Machine Learning is the subset that gets there by learning from data rather than following rules a programmer wrote by hand. Deep Learning is the subset of ML built on multi-layered neural networks — the engine behind the last decade's breakthroughs. Generative AI is the slice of deep learning whose purpose is to create new content. And LLMs are generative models that specialize in language — one type of "foundation model."

Why is it called "generative"?

The name draws a line between the two great families of machine-learning models. The distinction is classic and was crisply framed by Ng & Jordan.

Question it answersDiscriminativeGenerative
What is it doing?Separating / labeling existing dataCreating new data
Typical task"Is this email spam?" "Cat or dog?""Write a poem." "Generate an image."
What it modelsThe boundary between classesHow the data itself is produced
ExamplesSpam filters, image classifiersLLMs, image generators (GANs, diffusion)
So, why "generative"?

Because the model generates — it produces brand-new content rather than just labeling what already exists. Remember Section 2's next-token prediction: an LLM writes the next word, then the next, building original text that was never in its training data. That act of creation is exactly what makes it generative AI.

Where does Agentic AI fit?

Here's the key insight that confuses many beginners: agentic AI is not a new kind of model. There is no separate "agent neural network." Agentic AI is a system pattern that takes a generative model — usually an LLM — and wraps it with the ability to plan, remember, and use tools, so it can take actions and pursue a goal instead of just returning one block of text.

Your roadmap from here

This tutorial climbs that ladder in order: LLMs (the generative brain, Section 2), embeddings (Section 3) and RAG (Section 4) that ground it in facts, then agents (Section 5) and multi-agent orchestration (Section 6) that give it hands and autonomy — closing with real-world use cases (Section 7).

Section 02

The Origin of LLMs

Before agents, before RAG, there was the language model. To understand everything that follows, you need a clear mental model of what a Large Language Model (LLM) actually is — and, crucially, what it is not.

What is a Large Language Model?

A Large Language Model is a neural network trained on enormous amounts of text to do one deceptively simple thing: predict the next token (a word or word-fragment) given the text so far. By learning to predict the next token across trillions of words, the model implicitly learns grammar, facts, reasoning patterns, and writing styles. The landmark GPT-3 paper showed that once such a model is large enough, it can perform many tasks it was never explicitly trained for, simply from a few examples in the prompt.

Core idea

An LLM is a probabilistic next-token predictor. It does not "look up" answers in a database. It generates the most statistically plausible continuation of your text, one token at a time. This single fact explains both its magic and its flaws.

How are they built?

Modern LLMs are built on the Transformer architecture, introduced in the 2017 paper "Attention Is All You Need." Its key innovation — self-attention — lets the model weigh the relevance of every word to every other word in a sequence, in parallel, instead of reading strictly left-to-right. This made it possible to train far larger models on far more data.

Building an LLM happens in stages:

1. Pre-training

The model reads a huge corpus (web pages, books, code) and learns next-token prediction. This is where raw knowledge and language ability come from. Scaling laws show that performance improves predictably as you grow model size, data, and compute.

2. Supervised fine-tuning

The pre-trained model is further trained on curated examples of instructions and good answers, teaching it to be a helpful assistant rather than just an autocomplete engine.

3. RLHF alignment

Reinforcement Learning from Human Feedback uses human preference ratings to make answers more helpful, honest, and harmless. This is the step that produced models like InstructGPT and ChatGPT.

How do they actually work?

When you send a prompt, four things happen under the hood:

① Tokenization

Your text is split into tokens — common words or sub-word fragments — using algorithms like Byte-Pair Encoding. "tokenization" might become token + ization. Models think in tokens, not letters.

② Embeddings

Each token becomes a high-dimensional vector — a list of numbers capturing meaning. Tokens with similar meanings sit close together in this "vector space." (We reuse this idea heavily in RAG.)

③ Attention

Stacked Transformer layers use self-attention to let every token gather context from every other token, building a rich representation of what your prompt means.

④ Next-token sampling

The model outputs a probability for every possible next token. One is sampled (the temperature setting controls how "creative" vs. deterministic this is), appended, and the loop repeats.

Two limits worth remembering

Context window: a model can only "see" a fixed number of tokens at once — everything outside that window is invisible. Knowledge cutoff: a model only knows what was in its training data, frozen at a point in time. It has no built-in awareness of today's news, your private documents, or live data. RAG (Section 4) and tools (Section 5) exist largely to overcome these two limits.

Why do they hallucinate?

A hallucination is when a model produces text that is fluent and confident but factually wrong or unsupported. This is not a random bug — it is a direct consequence of how LLMs work. Because the model's job is to generate plausible next tokens, it will happily fabricate a plausible-sounding citation, statistic, or API that never existed, because that fabrication is statistically likely text.

Researchers categorize and explain hallucination in several ways:

  • It optimizes for plausibility, not truth. The model has no internal "fact checker"; nothing forces a generated statement to be grounded in reality.
  • Gaps and noise in training data. If knowledge is missing, outdated, or contradictory in the training set, the model fills the gap with a guess.
  • Training and evaluation reward confident guessing. Recent work argues that standard training/evaluation procedures effectively reward the model for guessing rather than saying "I don't know."
Why this matters for everything ahead

Hallucination is the central problem that motivates the rest of this tutorial. RAG reduces it by grounding the model in retrieved facts. Agents reduce it further by letting the model verify its work with real tools instead of guessing. Keep this thread in mind as you read on.

Quick quiz — check yourself

  1. What does an LLM literally compute at each step?
    Show answer

    A probability distribution over its whole vocabulary for the next token — then it samples one token from that distribution and repeats.

  2. What is a token, and why don't models work in whole words or single letters?
    Show answer

    A token is a sub-word fragment (from an algorithm like Byte-Pair Encoding). It balances vocabulary size against sequence length — words would need a huge vocab; letters would make sequences far too long.

  3. Why do LLMs hallucinate?
    Show answer

    They are optimized to produce plausible next tokens, not true ones. With nothing grounding them in verified facts, a fluent-but-wrong continuation can be the highest-probability path.

  4. What does temperature change about the distribution?
    Show answer

    It scales the logits before softmax: low temperature sharpens the distribution (deterministic, favors the top token); high temperature flattens it (more random/creative).

Section 03

Embeddings, Explained Simply

A computer can't read. So how can it possibly know that "happy" and "joyful" mean almost the same thing, while "happy" and "brick" have nothing to do with each other? The answer is one of the most beautiful ideas in AI: the embedding. It's the bridge between everything in Section 2 and the RAG and agents that follow — so let's build the intuition from scratch.

The one-sentence idea

An embedding turns a word (or sentence, or document) into a list of numbers — a set of coordinates — so that things with similar meaning end up close together, and a computer can finally measure "how alike are these two ideas?"

Think of it as a giant map of meaning

Imagine an enormous map. Instead of cities, every word gets its own pin. Here's the rule for placing pins: words that mean similar things go close together; unrelated words go far apart. So cat, kitten, and dog cluster in one neighborhood; apple, banana, and grape sit together in another; and king and queen live near each other somewhere else entirely.

Nobody draws this map by hand. The model figures out where each pin goes by reading a mountain of text and noticing which words keep similar company. As the linguist J. R. Firth put it in 1957, "You shall know a word by the company it keeps." If two words show up in the same kinds of sentences ("I fed the ___," "the ___ purred"), the model learns they belong near each other. This is exactly how the famous word2vec and GloVe systems learned their maps.

An embedding is just the coordinates — and you can do math with meaning

On a real-world map, a pin's location is two numbers: how far east, how far north. An embedding is the same idea — a list of numbers giving a word's position on the meaning map. The only twist is that the meaning map needs many directions (one for "is it an animal?", one for "is it royal?", one for "is it positive or negative?", and so on), so a real embedding isn't 2 numbers — it's often hundreds or thousands of them. We just can't draw that, so we flatten it to 2-D to picture it.

Here's the part that feels like magic. Because words are now numbers, you can literally do arithmetic with meaning. The result that stunned researchers in 2013:

Meaning, as an equation

king − man + woman ≈ queen

Start at king, subtract the idea of man, add the idea of woman — and you land right next to queen. The model was never told this. It fell out of the map it learned on its own. This is the look-up that word2vec's authors documented.

How a computer measures "are these alike?"

Once everything is a point in space, similarity becomes simple geometry: two words are alike if their pins point in nearly the same direction from the center. Computers measure the angle between two embeddings (called cosine similarity) — a small angle means very similar, a wide angle means unrelated. That single number is the workhorse behind search, recommendations, and RAG.

# Words become vectors; "closeness" = similar meaning
embed("happy")  # → [0.21, -0.04, 0.88, ...]  (hundreds of numbers)
embed("joyful") # → [0.19, -0.02, 0.85, ...]  ← almost the same direction
embed("brick")  # → [-0.77, 0.40, 0.05, ...]  ← points somewhere else entirely

similarity("happy", "joyful")  # ≈ 0.94  (very close)
similarity("happy", "brick")   # ≈ 0.05  (basically unrelated)

Why embeddings matter (and where we use them next)

It's not just single words. We can hand a whole sentence or document to an embedding model and get one set of coordinates that captures its overall meaning — that's what Sentence-BERT made practical. That unlocks a superpower: to find text that means the same thing as your question — even if it uses completely different words — you just look for the nearest pins on the map.

🔍 Semantic search

Find results by meaning, not exact keywords. "How do I reset my password?" matches "account recovery steps."

📚 RAG

The engine of the next section: embed your question, fetch the nearest document chunks, ground the answer in them.

🧠 Agent memory

Store past facts as embeddings so an agent can later recall the relevant ones — long-term memory (Section 5).

Carry this forward

Everything you're about to read in RAG and agent memory is built on this one idea: turn meaning into coordinates, then find what's nearby. If embeddings click for you, the rest of this tutorial will feel like a natural next step.

Quick quiz — check yourself

  1. What does cosine similarity actually measure?
    Show answer

    The angle between two vectors. ~1.0 = pointing the same way (very similar meaning), ~0 = unrelated. It ignores vector length, so it compares direction (meaning) only.

  2. Two sentences share almost no words yet score 0.8. How is that possible?
    Show answer

    Embeddings encode meaning, not surface words. Paraphrases land in nearly the same place in vector space even with different vocabulary.

  3. What does king − man + woman ≈ queen demonstrate about embedding space?
    Show answer

    That relationships are captured as consistent directions — you can do arithmetic with meaning. The model learned this structure on its own.

  4. UMAP squashes 1536-D vectors to 3-D for the plot. What is the risk of trusting that picture?
    Show answer

    Reduction is approximate — it preserves neighborhoods but distorts exact distances and global layout. Clusters are real; precise positions and gaps between them are not literal.

Section 04

The Art of Grounding the LLM — Retrieval-Augmented Generation

If an LLM hallucinates because it relies only on frozen, parametric memory, the fix is intuitive: give it the right facts at the moment it answers. That is exactly what Retrieval-Augmented Generation (RAG) does.

An overview of RAG

Retrieval-Augmented Generation was introduced by Lewis et al. in 2020. The idea: instead of relying solely on what the model memorized during training, retrieve relevant documents from an external knowledge source at query time and feed them into the prompt as context. The model then generates its answer grounded in that retrieved evidence.

The one-sentence definition

RAG = "open-book exam" for an LLM. Rather than answering from memory alone (closed-book), the model is handed the relevant pages first, then asked to answer using them.

How does RAG reduce hallucination and ground the model?

Grounding works because the answer is now conditioned on supplied evidence rather than generated from the model's parametric guesses alone. Empirically, augmenting generation with retrieval has been shown to measurably reduce hallucination — Shuster et al. demonstrated that retrieval augmentation reduces hallucination in knowledge-grounded conversation while improving answer accuracy. Grounding gives three concrete benefits:

🎯 Accuracy

Answers are anchored to real, retrievable source text instead of statistical guesses.

🔗 Attribution

You can show which document a claim came from — enabling citations and trust.

🔄 Freshness

Update the knowledge base, not the model. Private and real-time data become usable without retraining.

How does RAG work? The ingestion → retrieval → generation pipeline

A RAG system has two phases. Ingestion happens ahead of time (offline): you prepare your knowledge base. Retrieval & generation happen live (online): each time a user asks a question.

The mechanics: embeddings & vector search

The engine of retrieval is the embedding: a model converts each chunk of text into a vector so that semantically similar text lands near it in vector space. Dense Passage Retrieval showed that learned dense embeddings dramatically outperform old keyword search for question answering. At query time we embed the question the same way and find the nearest chunks. Because comparing against millions of vectors exactly is slow, vector databases use Approximate Nearest Neighbor (ANN) algorithms such as HNSW graphs to find the closest matches in milliseconds.

# A minimal RAG loop in pseudocode
# --- Ingestion (run once) ---
chunks = split_into_chunks(documents)
vectors = embed(chunks)
index = VectorDB(vectors, metadata=chunks)

# --- Per query (run live) ---
def answer(question):
    q_vec   = embed(question)
    top_k   = index.search(q_vec, k=5)        # nearest chunks
    context = "\n\n".join(top_k)
    prompt  = f"Use ONLY this context:\n{context}\n\nQ: {question}"
    return llm(prompt)                       # grounded generation

Chunking & retrieval strategies

RAG quality lives or dies on retrieval quality, and retrieval quality starts with chunking — how you split documents before embedding. Chunks that are too big dilute relevance; too small and they lose context.

Chunking strategies

StrategyHow it worksBest for
Fixed-sizeSplit every N tokens/characters, often with overlap.Simple, uniform text; a fast baseline.
RecursiveSplit on a hierarchy of separators (paragraphs → sentences → words) to keep natural units intact. The standard LangChain RecursiveCharacterTextSplitter.Most prose; the sensible default.
SemanticSplit where the topic shifts, detected via embedding similarity between sentences.Documents with mixed topics.
ContextualPrepend a short, LLM-generated summary of the document to each chunk before embedding. Anthropic's Contextual Retrieval shows this sharply cuts retrieval failures.Large corpora where chunks lose context.

Retrieval strategies

Dense (semantic)

Match by meaning using vector similarity. Great for paraphrases and concepts, but can miss exact keywords, codes, or names.

Hybrid search

Combine dense vectors with classic keyword scoring (BM25) to get both meaning and exact-term matching — the current best-practice default.

Re-ranking

Retrieve a broad set, then use a more precise cross-encoder model (e.g. BERT re-ranker) to re-order the top results for relevance before sending to the LLM.

A common production pattern

Hybrid retrieve → re-rank → contextual chunks. Cast a wide net with BM25 + dense search, tighten precision with a re-ranker, and ensure each chunk carries enough context to stand alone. This stack addresses the most common cause of RAG failure: the right answer was in the knowledge base but never made it into the prompt.

Section 05

The Birth of Agents

RAG gives an LLM better knowledge, but it still only does one thing: produce text in a single shot. An agent breaks that limit. It uses the LLM as a reasoning engine inside a loop — thinking, acting in the world, observing results, and trying again until a goal is met.

What is an agent?

An agent is best understood through a now-canonical formula. As Lilian Weng describes it, an LLM-powered autonomous agent is built from the LLM as the core "brain," surrounded by three faculties that give it agency: planning, memory, and tool use.

🧭 Planning

Break a big goal into steps and reason about how to achieve them. Chain-of-Thought prompting elicits step-by-step reasoning, and Reflexion lets an agent critique and improve its own attempts.

🧠 Memory

Short-term memory is the context window (the current conversation). Long-term memory stores facts in an external vector store the agent can retrieve later — RAG, used as memory.

🛠️ Tools

Tools let the model act: run code, query a database, call an API, search the web. Function calling and Toolformer showed models can learn to invoke tools reliably.

How agents augment the LLM — the ReAct loop

The breakthrough that made agents practical is ReAct ("Reasoning + Acting"). Instead of answering in one shot, the model interleaves Thought → Action → Observation steps: it reasons about what to do, takes an action with a tool, observes the real result, and repeats until done. This grounds its reasoning in real feedback instead of pure speculation.

What can agents do that plain LLMs cannot?

CapabilityPlain LLMAgent
Access live / private data❌ frozen at training cutoff✅ calls search, APIs, databases
Take real actions❌ only outputs text✅ sends emails, edits files, runs code
Multi-step tasks❌ single shot✅ loops until the goal is reached
Self-correction❌ can't see if it was wrong✅ observes results, retries, reflects
Precise computation❌ may miscalculate✅ delegates to a calculator/code tool

How do we develop agents?

At its heart, an agent is a loop around an LLM that can call tools. Here is the essential skeleton, stripped of any framework:

# The core agent loop — every framework is a wrapper around this
def run_agent(goal, tools, max_steps=10):
    history = [system_prompt(goal, tools)]
    for step in range(max_steps):
        decision = llm(history)                  # Thought + chosen Action
        if decision.is_final:
            return decision.answer              # goal reached
        result = tools[decision.tool](**decision.args)   # Act
        history.append(observation(result))      # Observe, then loop

In practice you rarely write the loop by hand. Mature building blocks include:

  • Tool use / function calling — the model returns structured calls your code executes (OpenAI function calling, Anthropic tool use).
  • The Model Context Protocol (MCP) — an open standard from Anthropic for connecting agents to tools and data sources in a uniform way, so you don't rebuild integrations per model.
  • Frameworks — LangGraph, AutoGen, and the OpenAI Agents SDK provide loops, memory, and orchestration out of the box (covered next).
Design guidance from the field

Anthropic's Building Effective Agents stresses: start with the simplest thing that works. Many "agent" problems are solved better by a fixed workflow than by a fully autonomous loop. Add autonomy only when the task genuinely requires open-ended decision-making — it costs latency, money, and predictability.

Section 06

Multi-Agent Orchestration

One agent is powerful; a coordinated team of agents can tackle problems too large or too varied for any single context. Orchestration is the discipline of arranging how agents — and humans — pass work between each other. The central design question is always: how much autonomy do we grant, and who holds the reins?

What is agency and multi-agent orchestration?

Agency is the degree to which the system — rather than a human — decides what to do next. Multi-agent orchestration is coordinating several specialized agents (each with its own role, prompt, and tools) toward a shared goal. A common and effective shape is the orchestrator-worker pattern: a lead agent decomposes a task and delegates subtasks to worker agents running in parallel. Anthropic reports this architecture outperformed a single agent by 90% on their internal research evaluation.

The spectrum of autonomy — three patterns of orchestration

Orchestration is not all-or-nothing. It sits on a spectrum from fully human-directed to largely self-directed. Your three core patterns map cleanly onto this spectrum:

Pattern 1 — Deterministic Workflow

The human weaves the entire flow, which is deterministic. Every step and transition is fixed in code; the LLM is called at predefined points to do bounded sub-tasks (classify, extract, summarize). This corresponds to Anthropic's "workflows," where LLMs are orchestrated through predefined code paths. It is the right choice whenever the task is well-understood — it is predictable, testable, and easy to debug.

Pattern 2 — Controlled Flow Engineering

Agents get autonomy, but in a controlled fashion woven by the human. The human designs the overall graph, but at certain nodes an LLM decides which branch to take, which worker to dispatch, or how many parallel paths to spawn. This covers the canonical building blocks — routing, parallelization, and orchestrator-workers — and is what frameworks like LangGraph are built to express.

Pattern 3 — Controlled Autonomy

Agents weave their own path, but the decision boundary is still controlled — via scoped tools, prompts, and permissions. Here the agent plans dynamically (a true ReAct loop), but humans constrain the sandbox: which tools exist, what they're allowed to touch, budget/step limits, and approval gates for risky actions. This is the most capable and the most demanding to govern — autonomy is real, but bounded by design.

The canonical orchestration building blocks

Anthropic's Building Effective Agents catalogs five composable patterns that underlie almost every real system. They are the vocabulary every practitioner should know:

    Show answer

    Advantage: specialization and parallelism. Risk: errors propagate between agents, plus higher cost and coordination complexity.

  4. Where in an orchestrated system should a human-in-the-loop checkpoint go?
    Show answer

    Right before any irreversible or high-stakes action (sending money, deleting data, publishing), so a person can approve it.

Section 07

The Use Cases of Agentic AI

Everything so far — LLMs, grounding, agency, orchestration — exists to solve real problems. Here are the most impactful, proven use cases, each with the flow that makes it work.

1 · Software engineering agents

Coding agents resolve real GitHub issues end-to-end: read the codebase, plan a fix, edit files, run tests, and iterate until green. The SWE-bench benchmark measures exactly this on real-world issues, and SWE-agent showed that giving an LLM a well-designed agent–computer interface dramatically improves its success rate. Tools like Claude Code apply this in production.

2 · Deep research agents

A research agent answers a broad question by planning sub-questions, searching many sources in parallel, reading and synthesizing, and producing a cited report. This is the orchestrator-worker pattern in production: a lead agent spawns parallel sub-agents, each researching one angle.

3 · Customer support agents

A support agent resolves tickets autonomously: it understands the request, retrieves the answer from a knowledge base (RAG), takes real actions through tools (issue a refund, update an order), and escalates to a human when it hits its boundary. Anthropic highlights customer support as a high-value agent application because it pairs conversation with concrete, tool-driven resolution.

4 · Computer-use agents

Computer-use agents operate a real graphical interface the way a person would: they look at the screen, move the cursor, click, and type to complete tasks across arbitrary software. Anthropic introduced this capability publicly with Claude's computer use, where the model perceives a screenshot and emits mouse/keyboard actions in a loop.

5 · Data & analytics agents

An analytics agent turns a plain-English question into an answer over real data: it inspects the schema, writes a query (SQL), runs it, checks the result, and explains it — self-correcting if the query errors. This is the ReAct loop applied to databases, and it makes data accessible to non-experts while keeping a human-defined boundary around what the agent may read or write.

The common thread

Notice the same skeleton in every use case: an LLM brain, grounded by RAG/tools, looping through act-and-observe, inside a human-defined boundary. That is the entire arc of this tutorial — from a next-token predictor to a goal-directed agent — applied to a domain.

Lab 1 — Watch the model predict (token probabilities)

An LLM never "decides" a word; at every step it produces a probability distribution over its whole vocabulary and the next token is sampled from it (recall Section 2). OpenAI's logprobs option lets you read that distribution directly. A confident fact spikes onto one token; an open-ended prompt spreads across many — which is exactly why temperature and sampling make answers vary.

import math
from openai import OpenAI
client = OpenAI()

resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        # steer the chat model to CONTINUE the text, not reply to it (see note below)
        {"role": "system", "content": "You are an autocomplete engine. Continue the user's text directly."},
        {"role": "user", "content": "The capital of France is"},
    ],
    max_completion_tokens=1,  # only the very next token
    logprobs=True,         # ask for log-probabilities...
    top_logprobs=12,        # ...for the top 12 candidates (max 20)
)

for c in resp.choices[0].logprobs.content[0].top_logprobs:
    print(repr(c.token), round(math.exp(c.logprob), 4))   # logprob → probability
Sample output
Prompt: "The capital of France is"
  token             probability   distribution
  ' Paris'              99.20%   ██████████████████████████████
  ' the'                0.30%    █
  ' located'            0.12%    █

Prompt: "My favorite color is"
  token             probability   distribution
  ' blue'              42.10%    █████████████
  ' green'             18.700%   ██████
  ' the'               9.40%     ███
  ...spread across many plausible words
Chat model ≠ autocomplete

A chat model is trained to respond, not to continue your text. Send the bare fragment "The capital of France is" with no system prompt and the top next token is 'The' at ~99% — because the model is starting a full reply, "The capital of France is Paris." That's still correct next-token prediction! We add the autocomplete system prompt above so the next token genuinely becomes ' Paris' — the behavior this lab is trying to show.

The full script also prints each prompt's actual answer alongside its distribution, and then demonstrates the knobs that decide how a token is picked from that distribution:

🌡️ Temperature

Scales the distribution before choosing. Low (→0) sharpens it — almost always the single most likely token: focused, repeatable, great for facts and code. High (>1) flattens it — unlikely tokens get a real shot: creative but riskier. OpenAI range: 0–2.

🎯 Top-p (nucleus)

Keep only the most likely tokens whose probabilities add up to p, then sample from just those. top_p=0.1 = only the safest ~10% of probability mass; top_p=1.0 = no filtering. Tune temperature or top-p, not both.

🔢 Top-k

Keep only the k most likely tokens, drop the rest, sample from those (k=1 = always the best token). Note: OpenAI's API doesn't expose top-k — it offers temperature and top-p — but Anthropic, Gemini, and most local models do.

How temperature actually scales the numbers

The model's final layer emits a raw score for every possible token — its logits (call them z). To turn scores into probabilities it runs softmax. Temperature simply divides every logit by T before softmax, and that one division is the whole trick:

The softmax, with temperature

pi = exp(zi) / Σj exp(zj)  →  pi(T) = exp(zi / T) / Σj exp(zj / T)

Dividing by a small T (<1) stretches the gaps between logits, so the leader pulls further ahead — a spikier, more confident distribution. Dividing by a large T (>1) squashes the gaps, flattening everything toward uniform. As T→0 it becomes pure greedy (always the top token); as T→∞ it becomes a coin-flip across the whole vocabulary.

The script then takes one real distribution and shows you the numbers actually move — temperature scaling is exactly softmax(logprob / T) (log-probs work in place of logits — the constant offset cancels in softmax), while top-p and top-k zero out the tail and renormalize what's left:

Sample output — temperature reshaping "My favorite color is"
  token          T=0.25     T=1.0     T=2.0
  ----------------------------------------------
  ' blue'         95.2%     42.0%     23.4%   ← low T: top token dominates
  ' green'         4.0%     19.0%     15.8%
  ' red'           0.6%     12.0%     12.5%
  ' purple'        0.1%      8.0%     10.2%   ← high T: long-shots get a chance
  ...tail          0.1%     19.0%     38.1%

Lab 2 — Meaning as geometry (embeddings)

An embedding turns text into a vector so that similar meanings sit close together (Section 3). We embed five sentences, normalize them, and read off the cosine similarity between every pair. Paraphrases score high even when they share almost no words — the model matches meaning, not keywords.

import numpy as np
from openai import OpenAI
client = OpenAI()

sentences = ["The cat sat on the warm windowsill.",
             "A kitten napped in the sunny window.",
             "The stock market fell sharply today."]

resp = client.embeddings.create(model="text-embedding-3-small", input=sentences)
v = np.array([d.embedding for d in resp.data])      # (3, 1536) matrix
v /= np.linalg.norm(v, axis=1, keepdims=True)         # normalize → dot = cosine
print(v @ v.T)                                          # similarity matrix
Sample output
        S1    S2    S3
  S1   1.00  0.71  0.12
  S2   0.71  1.00  0.10     ← cat & kitten sentences: close (0.71)
  S3   0.12  0.10  1.00     ← unrelated topic: far (0.12)

The full script adds a stock-market paraphrase pair and auto-detects the closest pair: python labs/lab2_embeddings.py.

Lab 4 — See the whole map (UMAP)

A similarity number is convincing, but a picture is unforgettable. Embeddings live in 1536 dimensions, which we can't draw — so we use UMAP, an algorithm that squashes high-dimensional vectors down to 2-D/3-D while trying to keep neighbors as neighbors. We embed the very same sentences from Lab 2 and watch them organize themselves into topic clusters — the paraphrase pairs that scored "similar" as numbers now sit side by side as a picture.

import umap, numpy as np
from openai import OpenAI
client = OpenAI()

# the same 12 sentences Lab 2 measured, grouped pets / finance / food / travel
sentences = ["The cat sat on the warm windowsill.",
             "A kitten napped in the sunny window.",      # ← paraphrase pair
             "The stock market fell sharply today.",
             "Shares plunged on the exchange this afternoon.", # ← paraphrase pair
             # ...food and travel sentences too (see sample_data.py)]

resp = client.embeddings.create(model="text-embedding-3-small", input=sentences)
vecs = np.array([d.embedding for d in resp.data])     # (12, 1536)

Lab 3 — A complete tiny RAG pipeline

Now wire it together (Section 4): embed a small knowledge base, embed the question, retrieve the nearest chunks, and ask the model to answer using only that context. To prove retrieval actually matters, the knowledge base holds a made-up fact the model can't know — so we answer the same question twice and watch RAG turn a guess into a correct, source-backed answer.

import numpy as np
from openai import OpenAI
client = OpenAI()

kb = ["The Zephyr-9 was released by Aetheris Labs in March 2024.",
      "Zephyr-9 error code E7 means the water filter is missing.",
      "Aetheris Labs is headquartered in Lisbon, Portugal."]

def embed(texts):
    r = client.embeddings.create(model="text-embedding-3-small", input=texts)
    v = np.array([d.embedding for d in r.data])
    return v / np.linalg.norm(v, axis=1, keepdims=True)

kb_vecs = embed(kb)                                    # ingest once
q = "What does error code E7 mean on the Zephyr-9?"
scores = kb_vecs @ embed([q])[0]                       # cosine to every chunk
hits = [kb[i] for i in np.argsort(scores)[::-1][:2]]        # top-2 chunks

prompt = f"Answer using ONLY this context:\n{hits}\n\nQ: {q}"
ans = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": prompt}],
)
print(ans.choices[0].message.content)
Sample output
[ Without RAG ]  I'm not familiar with a "Zephyr-9", so I can't say
                 what error code E7 means.   ← honest, but useless

[ With RAG ]     Error code E7 on the Zephyr-9 means the water filter
                 is missing.                 ← grounded in retrieved context
The whole tutorial, in 40 lines

Same model, same question — retrieval is the only difference between a confident guess and a correct, source-backed answer. That is the engineer's move at the heart of trustworthy AI: don't ask the model to remember; give it the facts and ask it to reason. Run it: python labs/lab3_simple_rag.py.

Appendix A

Quick Glossary

Token

The unit an LLM reads and generates — a word or word-fragment.

Embedding

A numeric vector representing the meaning of text; similar meanings sit close together.

Context window

The maximum amount of text a model can consider at once.

Hallucination

Fluent but factually wrong or unsupported model output.

RAG

Retrieval-Augmented Generation — grounding answers in retrieved documents.

Vector database

A store optimized for finding the nearest embeddings to a query.

Agent

An LLM that loops — planning, using tools, and observing results — to reach a goal.

ReAct

The Thought → Action → Observation loop at the heart of most agents.

Tool / function calling

Letting a model invoke external code, APIs, or data.

MCP

Model Context Protocol — an open standard for connecting agents to tools/data.

Orchestration

Coordinating multiple agents (and humans) toward a shared goal.

Autonomy spectrum

From fixed human-designed workflows to bounded self-directed agents.
